# 충전기 교체 운영 가이드 (장애 / 노후 교체)

- **작성일**: 2026-05-08
- **대상**: 현장 기사, CS 운영자
- **관련 정책**: `/auths` idempotent re-provisioning (One-shot 정책 폐기)
- **관련 PR**: feat/auths_idempotent_reprovision

---

## 1. 핵심 원칙

> **같은 시리얼번호만 새 기기에 박으면, 추가 작업 없이 그대로 운영 재개됩니다.**

CS 포털에서 기존 등록 정보를 수정하거나 reset 요청을 보낼 필요 없습니다.

---

## 2. 표준 절차

### 단계 1 — 현장에서 충전기 회수
- 장애/노후 충전기를 회수합니다.
- 회수 기기의 시리얼번호를 메모해두면 추적이 쉽습니다 (필수 아님).

### 단계 2 — 새 기기 펌웨어 설정
새 충전기에 다음 두 가지만 일치시키면 됩니다:
- `시리얼번호` = 회수한 충전기와 **동일** (펌웨어 설정 또는 모뎀번호 등록)
- `x-channel` / `x-token` = 동일 제조사의 인증 토큰 (이미 펌웨어에 박혀 출고됐다면 그대로)

### 단계 3 — 현장 설치 후 부팅
새 기기가 부팅 시 자동으로 다음 흐름 수행:

1. `POST /auths` 호출 (`x-channel`, `x-token`, `origin`=시리얼, `model`)
2. CSMS 가 시리얼 매칭 → **기존 `clientId` 유지 + 새 `pwd` 발급**
3. 새 기기가 `clientId` + 새 `pwd` 로 OCPP WebSocket 접속
4. 구 기기의 OCPP 연결은 CSMS 가 자동 강제 종료
5. BootNotification → `Accepted` → 운영 재개

### 단계 4 — 정상 동작 확인
- CS 포털 **충전기 관리** 화면에서 해당 `clientId` 의 `lastHeartbeatAt` 갱신 확인
- 펌웨어 버전 / vendorName 등이 새 기기 정보로 업데이트됐는지 확인

---

## 3. 운영자가 신경 쓸 게 없는 이유

| 항목 | 어떻게 처리되는가 |
|---|---|
| 기존 `clientId` 유지 | 동일 시리얼이면 CSMS 가 자동 매칭 → 충전소 등록 / 결제 이력 / 세션 통계 모두 연속성 유지 |
| 비밀번호 회전 | `/auths` 호출 시점에 새 `pwd` 발급 → 구 기기 인증 즉시 실패 |
| 구 기기 OCPP 연결 | 새 기기 접속 시 `connectionManager` 가 구 WebSocket 자동 `terminate()` |
| 진행 중 충전 세션 | `BootNotification` 핸들러가 stuck 된 Active/Pending 세션을 자동 Stopped 처리 (`failReason: BootNotification:Reset`) |

## 4. 케이스별 가이드

### 4-A) 펌웨어 비휘발성 저장소 분실 (공장 초기화 등)

별도 조치 불필요. 다시 부팅하면 `/auths` 가 다시 호출돼 새 `pwd` 가 발급됩니다.

```bash
# 펌웨어 입장에서 실행되는 흐름
1. boot → 저장된 clientId/pwd 없음 인지
2. /auths 호출 → 200 OK 로 (clientId, pwd, wsUrl) 수신
3. 비휘발성 저장소에 저장
4. OCPP 접속 시작
```

### 4-B) 새 기기에 다른 시리얼 박아 출고됨

`/auths` 가 404 (시리얼 미등록) 반환. CS 가 새 시리얼을 사전 등록하거나, 펌웨어를 재플래시해 원래 시리얼로 맞추는 방식 둘 중 하나로 해결.

운영 권장: **같은 시리얼 박는 게 원칙**. 다른 시리얼이 박히면 운영 이력이 끊깁니다.

### 4-C) 같은 시리얼 박힌 기기 두 대가 동시 운영되는 의심

운영 흐름상 자연 발생 X. 만약 발생한다면:

- 두 기기가 모두 `/auths` 를 주기적으로 호출하면, 매번 새로운 한 쪽만 살아남고 다른 쪽은 인증 실패가 반복됩니다.
- CS 포털 로그에서 짧은 간격으로 `Re-provisioning completed` 가 반복 출력 → 즉시 알아챌 수 있음.
- 이 시점에 운영자는 한 기기를 회수하면 됩니다.

펌웨어 측 권장 구현: **`/auths` 는 비휘발성 저장소가 비었을 때만 호출**. 이후엔 저장된 자격증명 사용. 그러면 의심 케이스가 발생할 수 없습니다.

### 4-D) 구 기기를 다른 위치에 재배치하고 싶음

구 기기는 새 기기가 부팅한 시점에 자격증명이 무효화됐으므로, 다른 위치에서 사용하려면:

1. 구 기기의 비휘발성 저장소에서 `clientId/pwd/wsUrl` 삭제 (공장 초기화)
2. 새 시리얼을 박거나, CS 가 사전 등록한 다른 시리얼을 박기
3. 부팅 → `/auths` 호출 → 신규 자격증명 수신

## 5. 모니터링 지표

운영자가 주기적으로 보면 좋은 지표:

| 지표 | 위치 | 정상 범위 |
|---|---|---|
| `Re-provisioning completed` 로그 빈도 | `pm2 logs pvpentech-csms` 또는 Cloud Logging | 충전기 교체 빈도와 일치 (분당 0~1건 수준이면 정상) |
| 같은 stationId 의 `Duplicate connection — old connection terminated` 빈도 | `connectionManager.register()` 경고 | 교체 직후 1회만 발생, 반복되면 4-C 케이스 의심 |
| 동일 시리얼의 `/auths` 호출 빈도 | provisioning 로그 | 분당 1회 미만이 정상. 분당 수회 이상이면 펌웨어 재시도 루프 의심 |

## 6. 본 정책의 보안 고려사항

- **물리 접근 권한**: 같은 시리얼이 박힌 가짜 기기를 들고 와서 hijack 시도하려면 (1) 제조사 토큰 + (2) 실제 시리얼 번호를 모두 알고 있어야 함. 두 정보 모두 펌웨어에 박혀 있어 제조사·CS 외부에 노출되지 않으면 안전.
- **시리얼 추측 공격**: `EN[0-9]{7}` 같은 패턴은 추측 가능하나, `/auths` 단계에서 `x-token` 검증을 먼저 통과해야 하므로 차단됨. 토큰 없이는 시리얼만으로 자격증명을 받을 수 없음.
- **토큰 유출 시**: 그 채널의 모든 사전 등록 시리얼이 위험. 즉시 `manufacturer/{id}/regenerate-token` 으로 회전 + 펌웨어 OTA 업데이트 필요.
